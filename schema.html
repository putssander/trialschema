TrialSchema v1.0

TrialSchema is an open, portable representation of clinical-trial eligibility designed for automated patient matching. It captures the same information a clinician extracts when reading a trial protocol — what to check, where to look, and how to evaluate — in a shape that an AI agent can execute deterministically.

The primary consumer is not a human reader. It is an AI agent, matching engine, or workflow system that needs explicit trial logic, deterministic routing hints, and a stable wire format it can process end-to-end.

2. Design principles

  • Agent-first. The target runtime is an AI agent or matching system, so the format favors explicit execution semantics over narrative readability.
  • Domain-neutral. Oncology, cardiology, metabolic, rare disease — the format makes no assumptions about specialty or treatment modality.
  • Standards-aligned, not standards-bound. Codings follow FHIR shape (system+code+display); units use UCUM. But every place that accepts a code also accepts plain text — end users never have to map criteria to ontologies by hand.
  • Graceful degradation. When a constraint cannot be machine-evaluated, the extractor emits {"kind": "text", "description": …}. The matcher always has the verbatim source_text as ultimate fallback.
  • Decoupled what from where. A criterion's clinical concept (its concept) is independent of the document types the matcher should search (its routing).
  • Human-in-the-loop friendly. Every criterion has an enabled toggle, a rank, an editable clarified_text, and free-text guidance.
  • Minimal surface. Every field earns its place. Extension data goes in namespaced extensions bags.

4. Primitives

Coding

FHIR-shaped reference to a concept in a terminology. Every field is optional; text alone is a valid free-text fallback.

{ "system": "http://loinc.org", "code": "2160-0", "display": "Creatinine" }
{ "text": "ECOG performance status" }             // fallback when no code is known

Recommended systems: http://loinc.org (observations), http://snomed.info/sct (conditions/procedures), http://www.nlm.nih.gov/research/umls/rxnorm (medications), http://hl7.org/fhir/sid/icd-10, http://www.genenames.org (genes).

Quantity

{ "value": 50, "unit": "mL/min", "system": "http://unitsofmeasure.org", "code": "mL/min" }

UCUM is recommended. unit is the human-readable label; code is the canonical UCUM code (e.g. "a" for years, "mo" for months, "d" for days).

PartialDate

ISO-8601 date with optional precision: "2024", "2024-06", or "2024-06-15". Empty string means unknown.

6. Criterion

A single eligibility rule. The shape decouples what the criterion is about (concept), how to evaluate it (constraint, assertion), and where to look (routing).

{
  "id":             "INC-03",
  "kind":           "inclusion",           // inclusion | exclusion
  "enabled":        true,
  "rank":           3,                       // 1..N run order
  "cost_tier":      1,                       // optional 1..5 hint

  "source_text":    "Creatinine clearance ≥ 50 mL/min within 14 days of screening",
  "clarified_text": null,                    // optional LLM rewrite

  "concept": {
    "domain": "observation",           // see Concept domains below
    "code":   { "system": "http://loinc.org", "code": "2164-2", "display": "Creatinine clearance" },
    "text":   "Creatinine clearance"
  },

  "assertion":  "present",                  // present | absent | unknown
  "constraint": {
    "kind":     "comparison",
    "operator": ">=",
    "quantity": { "value": 50, "unit": "mL/min", "system": "http://unitsofmeasure.org", "code": "mL/min" }
  },
  "time_window": { "anchor": "screening", "before": { "value": 14, "unit": "day", "code": "d" }, "recency": "most-recent" },

  "routing":  { "primary": ["core-lab", "intake-notes", "mdt-notes"], "fallback": [] },
  "guidance": "",

  "provenance": { "extracted_at": "2026-05-21T12:00:00Z", "extracted_by": "openai/gpt-…" }
}

Concept domains

High-level clinical category. Tells the matcher which kind of patient data to fetch.

demographics — age, sex, etc.
condition — diagnoses, problem list
observation — labs, vitals, scores
medication — drugs, prior therapy
procedure — surgery, intervention
adverse-event — toxicity history
genomic — mutations, biomarkers
consent — willingness, ICF
logistic — travel, language, etc.
other — anything else

Assertion

present — the patient must satisfy the constraint. absent — the patient must NOT satisfy it (e.g. "no history of"). unknown — no assertion. Combining this with kind: "exclusion" + assertion: "present" reads naturally as "patient must NOT have …".

7. Constraints (tagged union)

Every constraint is discriminated by kind. The text variant is the always-valid fallback.

comparison
{ "kind": "comparison", "operator": ">=", "quantity": { "value": 18, "unit": "years", "code": "a" } }
Operators: < · <= · = · >= · > · !=
range
{ "kind": "range", "low": { "value": 0 }, "high": { "value": 2 } }
Inclusive bounds. Either low or high may be omitted.
code-set
{ "kind": "code-set", "match": "any",
  "values": [ { "system": "http://snomed.info/sct", "code": "254837009" } ] }
match ∈ any · all · none. Patient matches if their codes intersect / contain / disjoint values.
boolean
{ "kind": "boolean", "value": true }
exists
{ "kind": "exists" }
Pure presence/absence. Combine with assertion.
text · always-valid fallback
{ "kind": "text", "description": "ECOG performance status 0 or 1" }
Use whenever the constraint cannot be safely structured. The matcher will defer to LLM judgement.

8. Routing

Routing has two layers. The top-level routing_profile defines the document types available in this export, the default scan set for most criteria, and the general agent visit order. Each criterion's routing then marks which of those documents are primary or fallback for that specific rule. Empty arrays mean no criterion-specific preference; use routing_profile.default_scan_set in default_visit_order.

{
  "routing_profile": {
    "id": "default-clinical",
    "document_type_system": "https://trialschema.org/v1/document-types",
    "default_scan_set": ["intake-notes", "referral-letters", "mdt-notes"],
    "default_visit_order": ["intake-notes", "referral-letters", "mdt-notes", "pathology"],
    "document_types": [
      { "id": "intake-notes", "label": "Intake Notes", "hint": "Demographics, admin rules, general medical history" },
      { "id": "referral-letters", "label": "Referral Letters", "hint": "External clinical history validation" },
      { "id": "mdt-notes", "label": "Multidisciplinary Team Notes", "hint": "Team or specialty board decisions and therapeutic strategy" },
      { "id": "pathology", "label": "Pathology", "hint": "Histology, tissue biomarkers, biopsies" }
    ]
  }
}
IdLabelTypical content
intake-notesIntake notesDemographics, admin rules, general medical history.
referral-lettersReferral lettersExternal clinical history validation.
mdt-notesMultidisciplinary Team NotesMultidisciplinary team / specialty board decisions, consensus reviews, therapeutic strategy.
pathologyHistology / pathologyMicroscopic properties, tissue biomarkers, biopsies.
imagingImaging / functional testingX-ray, CT, MRI, PET, US, plus EKG, EEG, echo, spirometry.
treatment-historyTreatment historyPrior systemic therapy, medication lines, washouts, surgery.
molecular-genomicAdvanced / genomic labsSequencing, molecular profiling, liquid biopsy.

Default system URI: https://trialschema.org/v1/document-types. Profiles may add custom ids; criterion routing ids must be declared in routing_profile.document_types.

9. Care path optional

A care path is the standard ordered sequence of clinical phases a patient with the trial's condition typically moves through (diagnosis → staging → treatment → follow-up). It is orthogonal to eligibility logic: it lets a matcher reason about where in a treatment journey a candidate sits, separately from whether they meet the criteria.

The pattern is universal — oncology pathways, radiation-oncology protocols, heart-failure / diabetes management programmes, transplant pathways, stroke-care pathways — but most early-phase or single-encounter trials don't need one. Populate care_path only when the trial maps to a recognised standard pathway; otherwise omit it.

FHIR alignment. CarePath aligns with FHIR R5 PlanDefinition (a template of an ordered set of activities), not CarePlan (which is a patient-instantiated plan). Each phase aligns with a PlanDefinition.action.

{
  "id":    "breast-cancer",
  "label": "Breast cancer",
  "concept": { "system": "http://snomed.info/sct", "code": "254837009" },
  "phases": [
    { "id": "phase-1", "label": "Diagnosis", "key_activities": ["Imaging", "Biopsy"] },
    { "id": "phase-2", "label": "Neoadjuvant therapy", "key_activities": ["Chemotherapy"] }
  ]
}

11. FHIR & standards alignment

TrialSchema v1 is a deliberate subset of HL7 FHIR R5 ResearchStudy with eligibility-focused additions. Every Trial-level element maps cleanly:

TrialSchema v1FHIR R5Notes
TrialResearchStudy1:1 conceptually.
id / identifiersResearchStudy.identifierPrimary id is the NCT number when present.
titleResearchStudy.title
phaseResearchStudy.phaseSame vocabulary, kebab-cased.
sponsorResearchStudy.associatedParty[role=sponsor]
conditionsResearchStudy.conditionCodings (SNOMED CT recommended).
interventionsResearchStudy.focusWith type annotation.
enrollment.targetResearchStudy.recruitment.targetNumber
lifecycle.statusResearchStudy.statusSame value set.
lifecycle.start_date / completion_dateResearchStudy.periodDate precision per PartialDate.
criteria[]ResearchStudy.recruitment.eligibilityv1 inlines criteria instead of referencing a Group.
care_pathPlanDefinitionOptional. care_path.phases[] ↔ PlanDefinition.action[]. Use when the trial maps to a recognised clinical pathway.
Coding / Quantity / IdentifierFHIR Coding / Quantity / IdentifierField names and shape preserved.

Other standards

  • UCUM — recommended for all unit codes (Quantity.code).
  • SNOMED CT — conditions, procedures (http://snomed.info/sct).
  • LOINC — observations, lab tests (http://loinc.org).
  • RxNorm / ATC — medications.
  • ICD-10 / ICD-11 — diagnoses.
  • HGNC — genes / molecular biomarkers.
  • OMOP CDM — concept.domain values (condition, observation, medication, procedure) align with OMOP domain ids.
  • HL7 CQL — the constraint tagged union is shaped so a CQL transpiler can map each variant to exists, comparison, or interval expressions.

12. Ingesting ClinicalTrials.gov

The TrialSchema UI accepts the CT.gov API v2 JSON shape directly. Drop a single study record, an array of studies, or a JSONL file of records into the upload box — the format is auto-detected.

What happens automatically:

  • NCT id, title, phase, sponsor, conditions, interventions, enrollment, lifecycle dates and status are pulled deterministically from protocolSection — no LLM round-trip.
  • The free-text eligibilityModule.eligibilityCriteria blob is parsed into structured Inclusion / Exclusion criteria by the LLM, with best-effort LOINC / SNOMED / RxNorm code hints attached to each criterion's concept.
  • Routing is set per criterion based on its concept domain.
// CT.gov v2 → TrialSchema v1 (deterministic field map)
protocolSection.identificationModule.nctId            →  trial.id
protocolSection.identificationModule.briefTitle       →  trial.title
protocolSection.designModule.phases[]                 →  trial.phase           // PHASE3 → "phase-3"
protocolSection.sponsorCollaboratorsModule.leadSponsor.name
                                                     →  trial.sponsor.name
protocolSection.conditionsModule.conditions[]         →  trial.conditions[].text
protocolSection.armsInterventionsModule.interventions →  trial.interventions[]
protocolSection.designModule.enrollmentInfo.count     →  trial.enrollment.target
protocolSection.statusModule.overallStatus            →  trial.lifecycle.status
protocolSection.statusModule.startDateStruct.date     →  trial.lifecycle.start_date
protocolSection.statusModule.primaryCompletionDateStruct.date
                                                     →  trial.lifecycle.primary_completion_date
protocolSection.statusModule.completionDateStruct.date
                                                     →  trial.lifecycle.completion_date
protocolSection.eligibilityModule.eligibilityCriteria →  trial.criteria[]      // LLM-structured

13. Versioning

format_version uses MAJOR.MINOR. Minor versions are strictly additive (new optional fields, new enum members marked as such). Major versions may make breaking changes; this is 1.0, the inaugural release. There is no prior version to be compatible with.

Implementers SHOULD reject documents with an unknown format value or with a format_version whose MAJOR they do not understand. They SHOULD ignore unknown fields within a known major version.
